## 一、專案概述

一款以 AI 為核心的健身 app。使用者輸入身體資料後，由 AI 依據目標、訓練頻率與可用器材生成課表；進健身房時 app 以任務清單形式帶著使用者逐項完成訓練並記錄；累積資料後由 AI 週期性回顧並調整下一階段課表。

**核心使用情境（閉環）**

```
輸入身體資料 → AI 排課表 → 進健身房跟著清單練 + 記錄 + 休息計時
   → 看身體數據進步 → AI 週期回顧 → 調整下週課表 → 循環
```

## 二、技術棧

| 項目 | 選擇 | 理由 |
|---|---|---|
| 前端框架 | Flutter | 跨平台，一次開發 |
| 狀態管理 | Riverpod | 資料變動連動多畫面，依賴關係處理乾淨 |
| 本地資料庫 | Drift (SQLite) | 健身資料高度關聯，離線優先（健身房常無網路） |
| AI 服務 | Gemini API（開發階段用免費層） | 有免費額度、多模態（讀照片）、文字分析足夠 |
| 後端 | 無（純本地） | 自用階段不需要；架構預留未來接雲端 |
| 登入系統 | 無 | 同上 |

## 三、架構原則（現在就要做對）

即使目前只有單一使用者，仍須為「未來給其他人使用」預留擴展。原則是**介面預留，實作從簡**——不過度設計，但不把架構寫死。

### 三層抽象

1. **AiService（AI 呼叫層）**
   將所有 AI 呼叫獨立成一層介面。今天底下接 Gemini，未來要換 Claude、加成本控制、加防呆，只改這一層，不動其他程式碼。

2. **Repository（資料存取層）**
   資料存取包一層。今天底下是本地 Drift，未來要加雲端同步（如 Supabase），UI 與商業邏輯都不用改。

3. **資料模型預留 userId**
   所有資料掛在一個 user 底下，目前 userId 為固定值。未來要多人時，資料結構不需大改。

### 設計慣例

- **AI 輸出一律走「AI 預填 → 使用者確認 → 存檔」**。所有 AI 結果（讀照片、排課表）都不直接信任，須經使用者過目確認。
- **防呆與隱私先留 TODO**。自用測試階段可略過，但須在程式中標記，未來上線前補上（見[第七節](#七上線前待辦自用階段先留-todo)）。
- **AI 輸出要求 JSON 格式**，以便存入資料庫並渲染為畫面。

## 四、功能清單

功能分為「第一圈（核心閉環）」與「第二圈（加值）」。先完成第一圈做出可用閉環，第二圈各項獨立加值、逐一補上。

### 第一圈：核心閉環

#### 1. Onboarding（首次設定）

- 手動輸入：身高、體重、年齡、性別、訓練經驗、目標（增肌／減脂／維持）
- 設定訓練頻率：一週幾天、每次多久
- （InBody 照片匯入屬第二圈，先以手動輸入為主）

#### 2. AI 排課表

- 輸入：身體資料 + 目標 + 訓練頻率（+ 第二圈時加入器材清單）
- AI 回傳 JSON 格式課表
- 課表動作含：動作名稱、目標組數 × 次數、**建議休息秒數**
- 排課邏輯參考：4 天→上下肢分化；3 天→推拉腿；2 天→全身
- AI 預填 → 使用者可改 → 存檔

#### 3. 訓練追蹤 + 記錄 + 休息計時（主場景）

進健身房當下最常開的畫面，以任務清單形式運作。

- 點「開始今天的訓練」→ 載入今日課表動作清單
- 動作依序排列，當前動作高亮
- 狀態區分：已完成 / 進行中 / 未開始
- 完成一組 → 輸入實際重量／次數 → 打勾
- **打勾後自動開始休息倒數**
  - 休息秒數來自課表的「建議休息秒數」，計時畫面可臨時修改
  - 可跳過、可延長
  - 倒數結束以震動提示（健身房吵，震動比聲音實用）
  - 須能在背景運作、鎖屏後時間到仍發通知（Flutter 本地通知）
- 動作所有組完成 → 自動跳下一個動作
- 全部完成 → 存成一筆 WorkoutSession
- 中途離開能接續（session 狀態須儲存）
- **同時保存「計畫」與「實際」**：課表寫臥推 60kg×5，實際做 55kg×5，兩者都存。此落差是週期回顧的關鍵原料。

#### 4. 身體數據追蹤

- 記錄體重 / 體脂 / 肌肉量歷史
- 趨勢圖表

### 第二圈：加值功能

#### 5. 訓練追蹤的臨時調整

- 跳過動作、換動作、改順序（機器被占用等情境）
- 牽涉「計畫被改動後如何儲存」，邏輯較複雜，故延後

#### 6. 器材管理 + 照片辨識

- 以「場地（Gym）」概念管理，可多組（家裡／公司／商業健身房）
- 拍照 → AI（Gemini Vision）辨識器材 → 人工確認＋補充
- 每場地存一份結構化器材清單
- 排課表時帶入「今天在哪練」對應的器材，AI 只用清單內器材排課

#### 7. InBody 照片辨識

- 匯入 InBody 報告照片 → Gemini Vision 讀取體脂率、肌肉量、內臟脂肪等
- 因機型／版面不同，辨識結果須人工確認／修改後才存

#### 8. 週期回顧調整（第一版 AI 核心邏輯）

- 累積一週訓練記錄 + 體組成變化
- 週末由 AI 回顧分析 → 重排下週課表
- 分析重點：進步停滯、計畫與實際落差、疲勞（RPE）、肌群平衡

#### 9. 飲食控制

- 設定每日熱量 / 三大營養素目標
- 記錄餐點
- （拍照辨識列為選配，準確度有限，當輔助）

## 五、AI 功能總覽

四個獨立的 AI 功能，建議各寫成獨立的 service 函式，各自管好輸入輸出格式。

| # | 功能 | 類型 | 所屬 | 輸出格式 |
|---|---|---|---|---|
| 1 | 讀 InBody 照片抽數字 | Vision | 第二圈 | JSON（數值欄位） |
| 2 | 讀健身房照片認器材 | Vision | 第二圈 | JSON（器材清單） |
| 3 | 生成課表 | 文字生成 | 第一圈 | JSON（課表結構） |
| 4 | 週期回顧調整 | 文字分析 | 第二圈 | JSON（調整後課表 + 說明） |

## 六、資料模型

```
UserProfile（身體資料、目標、訓練經驗、userId）
│
├─ BodyMetrics（體重／體脂／肌肉量歷史，含 InBody 匯入）
│
├─ Gym（場地）
│   └─ Equipment（該場地器材清單）
│
├─ WorkoutPlan（AI 生成課表，綁定某個 Gym）
│   └─ PlannedExercise
│       ├─ 動作名稱
│       ├─ 目標組數 × 次數
│       └─ 建議休息秒數
│
├─ WorkoutSession（一次訓練）
│   ├─ 狀態：進行中 / 已完成
│   ├─ 對應哪份課表的哪一天
│   └─ LoggedExercise（每個動作）
│       ├─ 計畫：目標組數 × 次數（來自課表）
│       ├─ 完成狀態：未開始 / 進行中 / 完成
│       └─ LoggedSet（實際每組：重量 / 次數 / RPE / 打勾）
│
└─ NutritionLog（每日飲食）
    └─ FoodEntry（單筆餐點 + 熱量營養素）
```

**重點**：每個 LoggedExercise 同時帶「計畫」與「實際」，追蹤時能對照，記錄時能存差異，供週期回顧分析。

## 七、上線前待辦（自用階段先留 TODO）

以下為「未來給其他人使用」前必須補上的項目，現階段自用可略過，但須在程式中標記，避免遺漏。

- [ ] **AI 建議防呆**：別人不像開發者看得懂，AI 偶爾給出危險建議（熱量算太低、新手排太重）。須加安全邊界檢查。
- [ ] **隱私處理**：他人身體數據用免費層會被拿去訓練模型，上線前須切換至付費層（付費層不拿資料訓練）。
- [ ] **強化人工確認流程**：InBody／器材辨識的確認步驟，因他人較不會主動修改而更形重要。
- [ ] **使用者系統**：將固定的 userId 接上真正的登入與多使用者管理。
- [ ] **雲端同步**：在 Repository 層底下接上雲端（如 Supabase）。

## 八、開發順序

原則：**先做不靠 AI 的地基，讓 app 能跑、能存資料，再一個一個接 AI**。每一步都能單獨測試，出錯好抓。

1. 專案初始化 + 三層架構骨架
2. UserProfile + 手動 Onboarding（先不接 AI）
3. 訓練追蹤 + 記錄 + 休息計時（純資料庫，先用課表假資料跑通流程）
4. 身體數據追蹤（記錄 + 趨勢圖）
5. 接第一個 AI 功能：生成課表（取代上面的假資料）
6. ——第一圈完成，已是可用閉環——
7. 器材管理 + 場地（先手動輸入）
8. Vision：器材照片辨識
9. Vision：InBody 照片辨識
10. 週期回顧調整
11. 訓練追蹤的臨時調整
12. 飲食記錄 + 熱量目標

> **註**：UI 策略——前期（步驟 2~4）先用 Flutter 樸素預設介面把功能跑通；功能穩定後再用 Claude Design 定設計語言、美化核心畫面，最後請 Claude Code 照設計實作。先讓骨架能跑，UI 是最後一層皮。
